# feat: 쪽지 / 알림 / 신고 / 1:1 문의 시스템 추가

submission-15에 구현된 메시징 시스템 전체를 Artlink-pre-redesign에 포팅하기 위한 명세 문서.

---

## 기존 상황

- 사용자(Artist · Gallery · Admin) 간 직접 소통 수단이 없었음
- Artist가 Gallery에 지원/문의하려면 외부 메일에 의존
- 운영 측의 사용자 알림 채널이 없어 지원 결과/승인 결과를 사용자가 직접 새로고침해서 확인해야 했음
- 부적절한 메시지를 신고할 수 없고, Admin이 운영 개입할 도구가 부재
- 고객센터(1:1 문의 / FAQ) 페이지가 없어 운영 질문이 코드 외부로 흩어져 있었음

---

## 추가한 기능

### 1. 쪽지(Message) 시스템 — 4단계 구조

- **Artist 측**: 갤러리 → 공모전 → 대화 제목(스레드) → 상세
- **Gallery 측**: 공모전 → 지원자 → 대화 제목(스레드) → 상세
- 데스크톱(`useIsMobile` 브레이크포인트 **1280px**): Finder 컬럼 뷰
- 그 미만(태블릿/모바일): **브레드크럼 누적 레이어 뷰** — 상위 선택이 위로 차곡차곡 쌓이고 크럼 클릭 시 해당 레이어로 복귀, 들여쓰기로 계층 시각화
- 4단계 컬럼/뷰 모두 `1단계 : 갤러리|공모전`, `2단계 : 공모전|지원자`, `3단계 : 대화 제목`, `4단계 : 상세` 라벨 표기
- exhibitionId 기반 그룹핑 — 일반 문의는 `exhibitionId=0` 가상 항목 (`exhibitionTitle="일반 문의"`), Artist 측은 `galleryId=-partnerId` 음수 키 사용
- 첨부파일 (이미지/파일) — JSON string으로 DB 저장, 메시지당 최대 5개

### 2. 알림(Notification) 시스템

- Navbar 우측 벨 아이콘 + 미읽음 카운트 뱃지 (30초 polling)
- 알림 드롭다운: 최근 10개 표시, 클릭 시 `linkUrl`로 이동 + 읽음 처리
- **외부 클릭 닫기**: `mousedown` 전역 핸들러 + `[data-notif-region]` 마커 어트리뷰트 기반
- 알림 타입: `APPLICATION_STATUS`, `NEW_APPLICANT`, `APPROVAL_RESULT`, `MESSAGE_SANCTION`

### 3. 메시지 신고(Report) 시스템

- 받은 메시지에 🚩 아이콘 → 신고 모달 (사유: PROFANITY/SPAM/INAPPROPRIATE/OTHER + 상세 텍스트)
- 중복 신고 차단: `@@unique([messageId, reporterId])`
- Admin이 처리: DISMISSED(반려) / ACTIONED(인정, 메시지 삭제 옵션)
- **신고한 메시지 마스킹**: 본인에게만 "🚩 신고한 메시지입니다"로 가려짐 (서버 측 치환, 클라이언트 우회 불가)
- **제재 메시지 마스킹**: ACTIONED 처리된 메시지는 모든 참여자에게 "🛡️ 제재로 가려진 메시지입니다"로 가려짐

### 4. 제재 알림(MESSAGE_SANCTION)

- Admin이 ACTIONED 처리하면 메시지 송신자에게 자동 알림 생성
- 알림 본문: `귀하의 메시지 "{subject}"에 제재 조치가 이루어졌습니다. (메시지 삭제됨) 사유: {adminNote}`
- `linkUrl = /messages?partner={receiverId}&exhibition={exhibitionId}&subject={baseSubject(Re 제거)}` 형태로 MessagesPage URL 핸들러가 4단계 상세 레이어까지 자동 진입
- deleteMessage 시에도 삭제 전에 알림 먼저 생성 (Cascade 삭제 회피)

### 5. 1:1 문의(Inquiry) + FAQ — 고객센터

- 사용자: 제목/내용으로 문의 작성 → Admin 답변 후 `status: ANSWERED`
- FAQ: Admin이 등록/수정/삭제, 누구나 열람 가능 (인증 불필요)
- `/support` 라우트로 단일 페이지에 통합

```prisma
model Inquiry {
  id        Int       @id @default(autoincrement())
  subject   String
  content   String
  status    String    @default("OPEN") // OPEN, ANSWERED
  reply     String?
  userId    Int
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  @@index([userId])
  @@index([status])
}

model Message {
  id           Int         @id @default(autoincrement())
  subject      String
  content      String
  read         Boolean     @default(false)
  senderId     Int
  sender       User        @relation("sentMessages", fields: [senderId], references: [id], onDelete: Cascade)
  receiverId   Int
  receiver     User        @relation("receivedMessages", fields: [receiverId], references: [id], onDelete: Cascade)
  exhibitionId Int?
  exhibition   Exhibition? @relation(fields: [exhibitionId], references: [id], onDelete: SetNull)
  attachments  String?     // JSON: [{url, name, type, size}]
  createdAt    DateTime    @default(now())
  reports      MessageReport[]

  @@index([receiverId, read])
  @@index([senderId])
  @@index([exhibitionId])
  @@index([createdAt])
}

model MessageReport {
  id         Int      @id @default(autoincrement())
  messageId  Int
  message    Message  @relation(fields: [messageId], references: [id], onDelete: Cascade)
  reporterId Int
  reporter   User     @relation("messageReports", fields: [reporterId], references: [id], onDelete: Cascade)
  reason     String   // PROFANITY, SPAM, INAPPROPRIATE, OTHER
  detail     String?
  status     String   @default("PENDING") // PENDING, DISMISSED, ACTIONED
  adminNote  String?
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  @@unique([messageId, reporterId])
  @@index([status])
  @@index([createdAt])
}
```

## API 명세

### 쪽지 (`/api/messages`)

| 메서드 | 엔드포인트 | 설명 | 인증 |
|--------|-----------|------|------|
| `GET` | `/messages/unread-count` | 미읽음 카운트 | ARTIST/GALLERY |
| `GET` | `/messages/recipients` | 보낼 수 있는 수신자 목록 | ARTIST/GALLERY |
| `GET` | `/messages/conversations` | 대화 그룹 (role별 다른 응답 구조) | ARTIST/GALLERY |
| `GET` | `/messages/thread/:userId?exhibitionId=` | 특정 상대와의 스레드 (자동 읽음 처리, 마스킹 포함) | ARTIST/GALLERY |
| `GET` | `/messages?tab=inbox\|sent` | 인박스/보낸함 (호환) | ARTIST/GALLERY |
| `GET` | `/messages/:id` | 단건 조회 + 읽음 처리 | ARTIST/GALLERY |
| `POST` | `/messages` | 쪽지 전송 | ARTIST/GALLERY |

**`POST /messages` 요청 바디:**
```json
{
  "receiverId": 3,
  "subject": "안녕하세요",
  "content": "...",
  "exhibitionId": 1,                  // optional, 일반 문의는 미포함
  "attachments": [                     // optional, 최대 5개
    { "url": "/uploads/x.png", "name": "x.png", "type": "image/png", "size": 12345 }
  ]
}
```

**`GET /messages/conversations` 응답 구조 (Artist):**
```json
{
  "role": "ARTIST",
  "galleries": [
    {
      "galleryId": 1,
      "galleryName": "서울 현대 갤러리",
      "ownerId": 3,
      "exhibitions": [
        { "exhibitionId": 0, "exhibitionTitle": "일반 문의", "lastMessage": {...}, "unreadCount": 1 },
        { "exhibitionId": 1, "exhibitionTitle": "...", "lastMessage": {...}, "unreadCount": 0 }
      ],
      "totalUnread": 1
    }
  ]
}
```

**`GET /messages/thread/:userId` 응답 메시지 마스킹 규칙:**
- `sanctioned: true` (ACTIONED 신고 존재) → 모든 참여자에게 `content="[제재로 가려진 메시지입니다]"`, `attachments=null`
- `reportedByMe: true` (본인이 신고) → 본인에게만 `content="[신고한 메시지입니다]"`, `attachments=null`
- 서버 측 치환이라 클라이언트 우회 불가

**수신자 제한:**
- Artist → 본인이 지원한 공모의 갤러리 owner만
- Gallery → 본인 갤러리 공모에 지원한 Artist만

### 신고 (`/api/reports`)

| 메서드 | 엔드포인트 | 설명 | 인증 |
|--------|-----------|------|------|
| `POST` | `/reports` | 메시지 신고 | ARTIST/GALLERY |
| `GET` | `/reports?status=` | 신고 목록 | ADMIN |
| `PATCH` | `/reports/:id` | 처리 (DISMISSED/ACTIONED) | ADMIN |

**`POST /reports` 요청:**
```json
{
  "messageId": 42,
  "reason": "SPAM",
  "detail": "광고성 메시지입니다."
}
```
- 본인이 보낸 메시지 신고 불가 (400)
- 대화 참여자만 신고 가능 (403)
- 중복 신고 차단 (409)

**`PATCH /reports/:id` 요청:**
```json
{
  "status": "ACTIONED",
  "adminNote": "...",
  "deleteMessage": true                // optional
}
```
- ACTIONED 시 메시지 송신자에게 `MESSAGE_SANCTION` Notification 자동 생성
- `linkUrl = /messages?partner={receiverId}&exhibition={exhibitionId}&subject={baseSubject}` 형태
- `deleteMessage=true`이면 메시지 삭제 (Cascade로 신고도 삭제됨)
- 알림은 메시지 삭제 **이전에** 먼저 생성

## 핵심 구현 디테일 (재구현 시 필수 체크)

### 1. 쪽지 reply 가드 — `threadExId === null` 사용

```typescript
// ❌ WRONG: 일반 문의(exhibitionId=0)를 falsy로 막음
if (!replyContent.trim() || !threadPartnerId || !threadExId) return;

// ✅ CORRECT: 0 허용
if (!replyContent.trim() || !threadPartnerId || threadExId === null) return;

// 전송 payload — 0일 때 키 자체 생략 (backend Zod positive() 충돌 방지)
sendMutation.mutate({
  receiverId: threadPartnerId,
  subject: `Re: ${selSubject}`,
  content: replyContent.trim(),
  ...(threadExId ? { exhibitionId: threadExId } : {}),
  ...(replyAttachments.length > 0 ? { attachments: replyAttachments } : {}),
});
```

### 사전 이해

이 플랜은 ArtLink에 "쪽지 / 알림 / 신고 / 1:1 문의" 시스템을 추가하는 작업이다.
CLAUDE.md와 architecture.md를 먼저 읽고 프로젝트 구조를 파악한 뒤 아래 순서대로 진행한다.

핵심 설계 결정:
- 쪽지는 exhibition 기반 4단계 그룹핑 (Artist: 갤러리→공모전→스레드→상세, Gallery: 공모전→지원자→스레드→상세)
- 일반 문의(공모전 미연관)는 `exhibitionId=null`로 저장하되, 클라이언트/conversations API에선 `exhibitionId=0` 가상 항목으로 표시
- 데스크톱 Finder 컬럼뷰 vs 모바일 브레드크럼 누적뷰는 단일 페이지 안에서 `useIsMobile(1280)`으로 분기
- 신고/제재 시 메시지 마스킹은 **반드시 서버측에서 content/attachments 치환** (클라이언트 우회 불가)
- 알림은 인증 사용자 누구나 받을 수 있는 범용 모델, `linkUrl`로 화면 이동
- 모바일 브레드크럼은 `useState` 누적 X, 현재 selL1/selL2/selSubject 상태로 매 렌더 계산
- TanStack Query는 모든 mutation에서 cross-component invalidate

+function baseSubject(s: string) { return s.replace(/^(Re:\s*)+/i, ''); }

+interface SubjectThread {
+  subject: string; messages: Message[]; lastDate: string; unreadCount: number;
+}
+
+function groupBySubject(messages: Message[], myId: number): SubjectThread[] {
+  const map = new Map<string, Message[]>();
+  for (const m of messages) { const b = baseSubject(m.subject); if (!map.has(b)) map.set(b, []); map.get(b)!.push(m); }
+  return Array.from(map.entries())
+    .map(([subject, msgs]) => ({
+      subject,
+      messages: msgs.sort((a, b) => new Date(a.createdAt).getTime() - new Date(b.createdAt).getTime()),
+      lastDate: msgs[msgs.length - 1].createdAt,
+      unreadCount: msgs.filter(m => m.receiverId === myId && !m.read).length,
+    }))
+    .sort((a, b) => new Date(b.lastDate).getTime() - new Date(a.lastDate).getTime());
+}
