# ArtLink 출시 전 종합 검수 리포트

검수일: 2026-05-30 · 대상: 로컬 서버(내 컴퓨터)에서 직접 실행·공격·자동 브라우저 테스트
※ 실제 서비스(artlink.cc)는 건드리지 않았습니다. 코드는 한 줄도 고치지 않았고, 분석·검증·리포트만 했습니다.

> **전제(이미 알고 계신 것):** 지금 로그인은 "개발용 퀵 로그인"입니다. 카카오 OAuth 승인 대기 중이라 임시로 쓰는 것이고, **출시 전 정식 로그인(OAuth/이메일)으로 교체 예정**입니다. 이 리포트는 그 교체를 전제로, 그 외 모든 것을 평가합니다. (로그인 관련 보안 항목은 맨 아래 "교체 시 챙길 것"에 따로 정리했습니다.)

---

## 📌 3줄 요약

1. **출시해도 되나?** → **네, 가능합니다.** 정식 로그인(OAuth) 전환만 끝내면 출시를 막을 치명적 문제는 없습니다. 나머지는 출시 후 다듬어도 되는 수준입니다.
2. **가장 급한 문제 1개** → **회사(사업자) 정보가 비어 있는 것.** 한국에서 통신판매중개 사이트를 운영하려면 사업자등록번호 등을 화면에 표시해야 하는데 지금 빈칸입니다. (코드 문제 아니고 내용만 채우면 됨)
3. **점수** → **10점 만점에 8점.** 기능·보안 뼈대가 튼튼하고 자동 테스트도 깔끔히 통과했습니다. 자잘한 사용자 경험(없는 페이지 안내, 오류 화면)만 보완하면 더 높아집니다.

## ✅ 잘 만들어진 점 (직접 테스트로 확인)

- **로그인 보안 자체는 견고합니다.** 가짜 신분증(위조 토큰)을 만들어 들이밀어도 서버가 다 막아냈습니다(401 거부). 로그인 안 하고 보호된 페이지에 들어가려 하면 자동으로 로그인 화면으로 보냅니다.
- **권한 분리가 정확합니다.** 작가 계정으로 관리자 전용 기능에 손대려 하니 딱 막혔습니다(403 거부).
- **남의 정보 훔쳐보기 방어됨.** 주소창 숫자만 바꿔 다른 사람 문의를 열려 했더니 "없음(404)" 또는 빈 목록만 나왔습니다.
- **비밀번호를 안전하게 저장합니다.** 그대로 저장하지 않고 bcrypt(알아볼 수 없게 뭉개는 표준 방식)로 처리합니다.
- **요청 폭주 방어 작동.** 로그인 시도를 빠르게 반복했더니 일정 횟수 후 자동 차단(429)됐습니다. 비밀번호 무차별 대입 공격을 막아줍니다.
- **악성코드 주입(SQL 인젝션)에 안전.** 공격 문자열을 넣어봤지만 Prisma(데이터베이스 안전 도구)가 막아 멀쩡했습니다.
- **비밀정보 노출 없음.** 100개 커밋 전체를 훑었지만 코드에 새어나간 비밀번호·열쇠가 없었습니다(gitleaks 0건).
- **세 역할 모두 정상 작동.** 자동 브라우저로 작가·갤러리·관리자 각각 로그인해 마이페이지를 열었더니, 역할에 맞는 메뉴가 정확히 떴고 **화면 오류가 단 한 건도 없었습니다.** 찜하기·로그아웃도 잘 됐습니다.
- **소유자 확인이 꼼꼼합니다.** 갤러리 수정·삭제·이미지 관리 등 거의 모든 기능에 "내 것이 맞나" 확인 코드가 들어 있습니다.
- **핵심 비즈니스 복합 시나리오를 끝까지 실제로 돌려 확인했습니다.** 갤러리 등록→관리자 승인→공개 노출, 공모 등록(커스텀 질문 포함)→승인→작가 검색 노출, 작가 지원(커스텀 답변)→중복 차단→갤러리 알림, 지원자 확인→상태변경(검토중→수락)→작가 알림, **리뷰 작성 시 별점 자동 재계산(5점→3점→삭제 시 0점 정확)**, 전시(Show) 등록→승인→노출, 수정요청→승인→실제 반영, 탐색 좋아요, 관리자 운영 CRUD(히어로·혜택·이달의 갤러리), 거절 시 사유 강제(없으면 400)까지 전부 정상. (실측: docs/raw/composite-flow-probe.log)
- **메시지/신고/문의 깊은 흐름까지 실제로 돌려 확인했습니다.** 작가→갤러리 메시지 전송(첨부파일 포함)·역할 위반 전송 차단·본인 메시지 신고 차단·중복 신고 차단까지 모두 정상. 특히 **신고하면 "[신고한 메시지입니다]"로 가려지고, 관리자가 제재하면 "[제재로 가려진 메시지입니다]"로 바뀌며 발신자에게 제재 알림이 가는 것**, **문의→관리자 답변→문의자 알림까지** 끝까지 작동했습니다. (실측: docs/raw/message-report-inquiry-probe.log, UI 스크린샷: docs/raw/e2e/msg-*.png)

> 한마디로: **뼈대가 잘 지어졌습니다.** 로그인 방식만 정식으로 바꾸면 출시 준비가 거의 끝납니다.

## 🔴 반드시 출시 전에 고쳐야 할 것

> **새로 발견된 출시 차단급 문제는 없습니다.** (단, 위 전제대로 "정식 로그인 전환"은 출시의 기본 조건입니다 — 맨 아래 "로그인 교체 시 챙길 것" 참고.)

법적으로는 아래 문제 1을 사실상 출시와 동시에 처리해야 합니다.

### 문제 1. 한국에서 운영하려면 표시해야 하는 사업자 정보가 비어 있습니다

- **[무슨 문제]**: 화면 맨 아래(푸터)에 "사업자등록번호, 통신판매신고번호, 주소, 대표번호" 칸이 다 빈칸입니다.
- **[실제로 벌어지는 일]**: "통신판매중개자"라고 스스로 밝힌 사이트는 법적으로 사업자 정보를 표시해야 합니다. 빈칸이면 신고·과태료 소지가 있고, 이용자 신뢰도 떨어집니다.
- **[얼마나 급한가]**: 🟡 출시 후 빨리 (법적 사안이라 사실상 출시와 동시에)
- **[내가 할 일]**: 코드 문제가 아니라 **내용 채우기**입니다. `frontend/public/terms/company-info.txt`에 실제 사업자 정보를 적으면 됩니다. Claude한테 **"company-info.txt에 들어갈 항목 양식을 알려줘"** 라고 하면 빈칸 목록을 받을 수 있습니다.

## 🟡 출시하고 나서 빨리 고칠 것

### 문제 2. 없는 주소로 들어가면 하얀 빈 화면만 나옵니다 (404 페이지 없음)

- **[무슨 문제]**: 사용자가 잘못된 주소를 입력하면 "페이지를 찾을 수 없습니다" 안내 대신 텅 빈 흰 화면이 뜹니다.
- **[실제로 벌어지는 일]**: 예를 들어 누가 오타 난 링크를 누르면, 아무것도 없는 흰 화면을 보고 "사이트가 고장났나?" 하고 나가버립니다. (자동 브라우저로 확인 → 본문 완전 공백, 스크린샷: docs/raw/e2e/03-unknown-route.png)
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"잘못된 주소로 들어오면 보여줄 404 안내 페이지를 만들고, 홈으로 돌아가는 버튼도 넣어줘"** 라고 하세요. (참고: frontend/src/App.tsx 라우트에 `path="*"` 없음)

### 문제 3. 없는 갤러리/전시를 열면 "없습니다" 안내 없이 거의 빈 화면입니다

- **[무슨 문제]**: 삭제됐거나 존재하지 않는 갤러리 주소를 열면 친절한 안내 없이 머리글만 있는 휑한 화면이 나옵니다.
- **[실제로 벌어지는 일]**: 예를 들어 오래된 링크를 누르면 "갤러리를 찾을 수 없습니다" 같은 안내가 없어 사용자가 당황합니다. (다행히 "계속 빙글빙글 로딩"되며 멈추는 현상은 **없었습니다**. 스크린샷: docs/raw/e2e/04-nonexistent-gallery.png)
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"갤러리·전시·공모 상세에서 데이터를 못 찾으면 '찾을 수 없습니다' 안내와 목록으로 가는 버튼을 보여줘"** 라고 하세요.

### 문제 4. 인터넷이 끊기거나 서버 오류가 나도 사용자에게 아무 안내가 없습니다

- **[무슨 문제]**: 목록 페이지에서 데이터 불러오기에 실패하면 오류 메시지 없이 빈 화면처럼 보입니다.
- **[실제로 벌어지는 일]**: 예를 들어 지하철에서 인터넷이 잠깐 끊긴 채로 갤러리 목록을 열면, "불러오지 못했습니다, 다시 시도" 안내가 없어 사용자는 사이트가 고장난 줄 압니다. (코드 검토로 발견 — 목록 페이지들에 실패 시 안내 화면이 없음. 강제로 끊어 재현하진 않아 "코드상 확인" 단계입니다)
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"갤러리·공고·전시 목록 페이지에 데이터 불러오기 실패 시 보여줄 오류 안내와 다시 시도 버튼을 추가해줘"** 라고 하세요. (참고: GalleriesPage / ExhibitionsPage / ShowsPage)

### 문제 5. 외부 라이브러리(부품)에 알려진 보안 취약점이 있습니다

- **[무슨 문제]**: 코드가 가져다 쓰는 외부 부품 중 일부가 구버전이라 알려진 약점이 있습니다. 특히 axios(통신 부품), multer(파일 업로드 부품)는 실제 서비스에 쓰입니다.
- **[실제로 벌어지는 일]**: 대부분 당장 악용되긴 어렵지만, 방치하면 공격에 이용될 수 있는 "문 안 잠근 창문"입니다. (npm audit 결과: 백엔드 9건·프론트 14건, 위험도 high 다수. 원본: docs/raw/npm-audit-*.json)
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"npm audit 결과 중 axios와 multer 등 실제 서비스에 쓰이는 부품을 안전한 버전으로 올려주고 테스트가 깨지지 않는지 확인해줘"** 라고 하세요.

### 문제 6. 업로드 파일을 "확장자 이름"으로만 검사합니다

- **[무슨 문제]**: 파일을 올릴 때 파일 이름 끝(.pdf 등)만 보고 통과시킵니다. 실제 내용물은 확인하지 않습니다.
- **[실제로 벌어지는 일]**: 예를 들어 위험한 파일의 이름만 `.hwp`로 바꿔 올리면 통과될 수 있습니다.
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"파일 업로드에서 확장자뿐 아니라 실제 파일 형식(MIME 타입)도 검사하도록 해줘"** 라고 하세요. (참고: backend/src/routes/upload.ts 파일 필터)

### 문제 7. 지원 상태를 거꾸로 되돌릴 수 있습니다 (순서 강제 없음)

- **[무슨 문제]**: 지원 상태(접수→검토중→수락/거절)가 정해진 순서 없이 아무 값으로나 바뀔 수 있습니다.
- **[실제로 벌어지는 일]**: 갤러리 담당자가 실수로 "수락"한 지원자를 다시 "접수"로 되돌려도 막지 않습니다. 데이터가 꼬일 수 있습니다. (피해 범위는 자기 공모 안으로 한정돼 크진 않음)
- **[얼마나 급한가]**: 🟡 출시 후 빨리
- **[내가 할 일]**: Claude한테 **"지원 상태 변경 시 정해진 단계 순서만 허용하고 역행은 막아줘"** 라고 하세요. (참고: backend/src/routes/exhibition.ts 지원 상태 변경)

## 🟢 천천히 고쳐도 되는 것

- **문제 8. 잘못된 값 입력 시 500 오류**: 숫자가 와야 할 곳에 글자를 보내면 친절한 "잘못된 요청(400)" 대신 "서버 오류(500)"가 납니다. → "입력값 형식이 틀리면 400으로 친절히 거절하도록 검증을 넣어줘". (확인됨: 글자 userId → 500)
- **문제 9. 모바일에서 알림창이 화면을 벗어날 수 있음**: 알림 드롭다운 폭이 고정(320px)이라 작은 폰에서 빠듯합니다. → "알림 드롭다운을 모바일 화면 폭에 맞게 반응형으로 바꿔줘". (참고: Navbar w-80)
- **문제 10. 작은 버튼 터치 영역**: 일부 수정/삭제 버튼이 손가락으로 누르기엔 작습니다(권장 44px 미만). → "리뷰 수정·삭제 버튼 등 작은 버튼의 터치 영역을 키워줘".
- **문제 11. 이미지 대체 텍스트 부족**: 일부 사진에 설명(alt) 텍스트가 비어 시각장애인 화면낭독기가 못 읽습니다. → "갤러리·리뷰·프로필 이미지에 의미 있는 alt 텍스트를 넣어줘".
- **문제 12. 인스타 피드 오류 시 빈 화면**: 인스타 연동이 실패하면 "불러오지 못했습니다" 없이 그냥 비어 보입니다(데이터 손실은 아님).
- **문제 13. 로그 파일 경로 처리(semgrep 경고)**: 내부 로그 저장 경로 처리에 이론상 약점 경고. 외부 입력이 직접 닿진 않아 위험은 낮음. (참고: backend/src/lib/logger.ts:25)
- **문제 15. 커스텀 질문의 "최대 선택 수"가 옵션 개수보다 커도 서버는 안 막음**: 화면(프론트)에서는 막지만, 서버에 직접 요청을 보내면 "옵션 2개인데 최대 5개 선택" 같은 모순된 공모도 생성됩니다(HTTP 201로 확인). 일반 사용자는 화면을 거치니 영향이 작지만, 서버에도 같은 검사를 넣는 게 안전합니다. → "공모 커스텀 질문 저장 시 maxSelect가 옵션 수를 넘으면 서버에서도 거절하도록 해줘". (참고: backend/src/routes/exhibition.ts customFieldSchema)
- **문제 16. 프로필 정보 조회(/auth/me)에 아바타(프로필 사진)가 빠져 있음**: 아바타를 바꾸면 저장은 되지만(HTTP 200), 이후 사용자 정보를 /me로 다시 불러오면 아바타 값이 응답에 없습니다. 평소엔 브라우저에 저장돼 유지되지만, 특정 새로고침 경로에서 프로필 사진이 사라져 보일 수 있습니다. → "GET /auth/me 응답에도 avatar를 포함하도록 해줘". (참고: backend/src/routes/auth.ts /me 라우트 select)
- **문제 14. 삭제된 대상에 대한 수정요청 승인 시 500 오류**: 갤러리/공모가 이미 삭제됐는데 그 수정요청을 관리자가 승인하면 친절한 안내 대신 "데이터 처리 중 오류"(500)가 납니다. 정상 경로(존재하는 대상)는 잘 동작합니다. → "수정요청 승인 시 대상이 없으면 500 대신 '대상을 찾을 수 없습니다' 안내를 주도록 해줘". (참고: backend/src/routes/approval.ts edit-request 승인 핸들러)
