# CLAUDE.md - ArtLink Full Technical Specification

이 문서는 ArtLink 플랫폼의 기술적 청사진입니다. AI 에이전트와 개발자는 시스템의 모든 기능을 아래 설계에 100% 충실하게 구현해야 합니다.

## 🛠 기술 스택 (Tech Stack)
- **프레임워크**: React + Vite + TypeScript
- **라우팅**: React-router v6
- **상태 관리**: TanStack Query v5 (Server), Zustand v5 (Client)
- **UI/스타일링**: Shadcn/ui, Tailwind CSS, Framer Motion (애니메이션)
- **폼/검증**: React-hook-form, Zod
- **인증**: Auth.js (개발 단계: Artist 1, Artist 2, Gallery, Admin 선택형 퀵 로그인)
- **백엔드/DB**: Express + Prisma + PostgreSQL
- **품질 관리**: ESLint, Prettier, Vitest (테스트)
- Backend와 Frontend를 분리해야함.
- 모바일웹 / PWA적용

## 🎨 유저 시나리오 및 UI 로직 상세

### 1. 초기 진입 및 공통 UI
- **Splash Screen**: 모바일 웹 구동 시 1초간 중앙 로고 노출 및 세련된 애니메이션 적용 필수.
- **Navigation Bar**: 전 페이지 노출. 좌측 [ArtLink 로고](클릭 시 홈 이동), 우측 [홈, 갤러리, 모집공고, 혜택, 마이페이지].
- **Hero Section (슬라이더)**:
    - Admin이 [사진, 제목, 링크 URL] 등록/수정/삭제 가능.
    - 3초 자동 슬라이드. **사용자가 수동 조작(슬라이드) 시 해당 시점부터 타이머 리셋 후 3초 뒤 다시 자동 시작.**
    - [바로가기] 버튼 우측 하단 배치: 외부 URL은 브라우저 새 창, 내부 URL은 라우팅 이동.
- **Center Catchphrase**: “갤러리와 아티스트를 잇다 : ArtLink” (중단 배치)
- **Quick Action Cards**: [갤러리 찾기 / 진행중인 공고 / 혜택] 페이지로 즉시 연결. 각각에 어울리는 픽토그램 생성.

### 2. 마이페이지 및 권한 (My Page)
- **인증**: 비로그인 시 로그인 창 노출. 로그아웃 시 로그인 페이지 이동. **프로필 사진 변경 기능** 포함.
- **Artist 유저**:
    - **포트폴리오**: [전시 이력, 작가 약력, 작품 사진(최대 30장)] 관리.
    - **찜 목록**: 갤러리(갤러리명)와 공모(갤러리명-공모명) 구분 노출. **찜 취소 시 목록에서 즉시 제거.**
    - **활동 내역**: 본인이 작성한 리뷰 및 지원한 공고 목록 모아보기.
- **Gallery 유저**:
    - **갤러리 등록**: [이름, 주소, 소개, 대표자명, 전화번호, 대표 이미지, 지역 태그] 승인 요청.
    - **공모 등록**: [대상 갤러리 선택, 제목, 구분(개인전/아트페어), 마감일, 전시일, 모집인원, 지역, 소개] 승인 요청.
    - **상태 확인**: 승인 대기 / 승인 거절(**거절 사유 확인 가능**) / 승인 완료 상태 제공.
    - **수정 정책**: 승인 후 상세 내용만 수정 가능. 초기 정보 수정은 Admin에게 수정 요청 기능 사용.
- **Admin 유저**:
    - **승인 큐**: 갤러리/공모 등록 및 수정 요청 승인/거절(**거절 시 사유 작성 필수**).
    - **운영 관리**: Hero Section, 혜택 목록 관리 시 **[등록 전 미리보기]** 기능 제공.
    - **이달의 갤러리**: 갤러리 검색 선정 및 **등록 기한 만료 시 자동 제거** 로직 구현.

### 3. 서비스 화면별 세부 동작
- **갤러리 찾기**:
    - 리스트: 1줄 1개 구성. 사진, 이름, 주소, 전화번호, 한줄소개, 찜하기, 별점 노출.
    - 필터: 지역(서울, 경기 북/남, 대전, 부산), 별점(3/4점+). **현재 적용된 필터 목록을 화면에 표시.**
    - 정렬: 별점순 나열 기능.
- **갤러리 상세**:
    - 상단 스와이프 사진, 우상단 하트(찜), 상세 소개(주인일 때만 수정 버튼 노출), 진행 중 공고(D-day).
    - 전시 종료 후 **홍보용 사진 및 후기 글 등록** 기능 (Gallery 유저 전용).
    - **리뷰**: Artist 전용. [별점, 사진(옵션), 텍스트]. 익명 시 `익명의 예술가 N` 표기. Admin은 삭제 버튼 노출.
- **모집 공고**:
    - D-day가 남은 공고만 노출. 갤러리 별점 및 지역 필터 제공.
    - **지원하기**: 클릭 시 마이페이지의 포트폴리오를 해당 갤러리 메일로 자동 전송 (Artist 전용).

## 📐 개발 및 신뢰성 원칙
- **사용자 가이드 작성**: 기능 구현 시, 해당 로직이 위치한 **코드 블록 위치를 명시한 개발자 가이드**를 주석이나 문서로 남길 것.
- **데이터 무결성**: 신규 등록/수정/삭제 시 서버 상태와 DB 동기화 무결성 보장 및 예외 처리 철저.
- **검증 및 테스트**:
    - 단일 기능 구현 후 Vitest 검증 필수.
    - **복합 시나리오 검증**: [등록 요청 -> Admin 승인 -> 검색 노출 -> 지원하기] 전체 흐름 테스트.
- **상업화 고려**: 환경 변수 및 인증 시스템(OAuth 등) 교체가 용이하도록 추상화된 구조 유지.
- **아키텍처** : 너가 작성한 아키텍처는 사람이 유지보수하기 용이해야하며, 참고할수있도록 항상 신규작성 및 변화가 있을경우 architecture.md에 저장 및 update해야해.

[구현시 주의 사항]
-개발 환경에서는 로컬서버를 띄워서 확인할 수 있도록 하지만, 추후 배포용으로 상업화 가능성을 두고 수정 가능하도록 구현해야함. 또한 로컬 서버에서 개발자가 확인을 위해 어떻게 웹을 띄울 수 있는지 가이드 및 ArtLink 폴더 하위에 run_web.sh로 자동실행 가능한 스크립트를 만들어야함.
-유지 보수가 가능하도록 간결하면서도 명확한 코드를 작성해야하고 주석 작성이 필요함.
- 유지보수를 위해서 너가 구현하면서 구현한 내용에 대한 사용자 guide를 작성해야함. 사용자 guide에는 개발자를 위해서 해당 내용이 있는 코드 block을 알려주면 좋음. (너무상세하게 모든곳에하면 토큰이 아까우니 큰 틀에서)
- 신규 갤러리 / 공모 / 리뷰 / 추가 및 수정/ 삭제 시에도 DB 및 프론트/백엔드 서버에 전혀 이상이 없이 작동할 수 있도록 해야함
- 단일 기능에 대한 구현 후 검증은 물론이고 여러 기능을 동시 및 순차적으로 실행했을 때에도 문제가 없이 자체검증할 계획을 세우고 검증에 통과할 때까지 개발을 해야함.
- 기능적인 부분은 물론이고 신뢰성 검증에도 중요성을 둬야함.

## ⚠️ 배포 규칙 (Render.com)
- **배포 URL**: https://artlink-2esp.onrender.com
- **배포 브랜치**: `deploy/render` (main에서 작업 → merge → push)
- **seed.ts upsert 규칙**: 스키마에 새 필드를 추가하면 seed.ts의 `update` 블록에도 반드시 해당 필드를 넣을 것.
  - Render DB는 유지되므로 기존 레코드는 update 경로를 탐. `update: {}`면 새 필드가 null로 남아 반영 안 됨.
  - 로컬에서는 `migrate reset`으로 DB를 날리고 create 경로를 타서 이 문제가 드러나지 않음.
- **PWA 캐시**: 배포 후 변경이 안 보이면 브라우저 DevTools → Application → Clear site data
